# LLM optimization challenges



**LITHE(LLM Infused Transformations of HEfty queries)**

The SQL queries embedded in enterprise applications are often riddled with inefficiencies and redundancies, especially when machine-generated by modeling software such as ORM tools

To prevent the LLM from "hallucinating" invalid or regressive queries, LITHE wraps the LLM in a strict, feedback-aware pipeline.





When LLM next-token confidence is low, multiple decision paths are explored. Monte Carlo tree search (MCTS) manages the practical overhead of this enumerative approach.



**Pipeline**

![](/assets/62-3-YujMGYEfwVhczSOyUZkQNmbXed9T4zbdcHmggE=.png)

QUas input and outputs a transformed query QT, together with (a) the *expected&#x20;*&#x70;erformance improvement of QT, in terms of optimizer estimated cost; (b) a *verification label&#x20;*&#x69;ndicating the semantic equivalence mechanism (provable or statistical) betweenQTandQU; and (c) a *reasoning&#x20;*&#x66;or why the LLM expects QT to be helpful wrt performance.

* **Stage 1: Prompt-based Rewriting & Syntax Verification:** The user's query is fed into the LLM using an ensemble of different prompts (processed in fresh, independent contexts). If the LLM generates a rewrite with syntax errors, LITHE uses the database engine's parser to catch the error and feeds the error message *back* to the LLM, prompting it to correct the mistake iteratively (up to a set threshold).
* **Stage 2: Query Costing:** To prevent performance regressions, syntactically valid rewrites are evaluated by the database's native optimizer. If the estimated abstract cost of the rewrite is higher than the original query, it is immediately discarded.
* **Stage 3: Fast Semantic Equivalence:** Checking if two complex SQL queries do exactly the same thing is mathematically hard. LITHE first uses a fast statistical method (correlated sampling) across multiple small, diverse sample databases. If the results match, it passes to the next stage.
* **Stage 4: Token Probability-driven Rewrite (MCTS):** This is where the system gets highly innovative. It models the rewriting process as a Markov Decision Process using **Monte Carlo Tree Search (MCTS)**. If the LLM is unsure about its next token (i.e., token probability falls below a certain confidence threshold \$\theta\$), the system explores multiple alternative "top-k" paths simultaneously to find the most optimal rewrite, explicitly preventing hallucinations.
* **Stage 5: Final Verification and Output:** The cheapest valid rewrite undergoes rigorous, logic-based formal verification using tools like QED or SQLSolver. It also undergoes heuristic checks for "brittleness" to ensure the optimizer isn't making false promises.



### Prompting Strategy Evolution



![](/assets/6P5nN_7YD2MYwUSYUPuoM_kmfhqW8vnTZw3PUCDiZPI=.png)

* Four-stage prompting approach with increasing sophistication
  1. Basic prompt: Simple rewrite request
  2. Explicit instructions: Maintain semantic/functional equivalence
  3. Detailed instructions: Human-level reasoning guidance
  4. Chain of thought: Step-by-step iterative instructions

LITHE was extensively evaluated on the rigorous **TPC-DS benchmark** (using a 100 GB database on PostgreSQL v16 with GPT-4o).

* **Speedups:** While the native optimizer and state-of-the-art (SOTA) automated tools struggle with hefty queries, the full LITHE system achieved a **10.57x** (up to 13.2x depending on the metric environment) geometric mean speedup. This performance closely rivals that of human experts (11.84x).
* **MCTS Impact:** The Monte Carlo Tree Search allowed the system to navigate out of low-confidence generations. In one highlighted example, exploring alternative rewrite paths when the model was uncertain yielded a massive **6,500x speedup**—a result that a standard zero-shot LLM prompt would have missed.
